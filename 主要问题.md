主要问题

 

 高优先级

 

1. Docker 后端镜像大概率构建失败

 

 backend/Dockerfile:13：

 

 COPY docker/backend/entrypoint.sh /entrypoint.sh

 

 但 Compose 里后端 build context 是 ../backend，见 docker/docker-compose.yml:60-63。

 实际文件在 docker/backend/entrypoint.sh，不在 backend/docker/backend/entrypoint.sh，所以 build context 内找不到。

 

2. Docker 生产前端无法代理 API

 

 前端硬编码 /api/v1：

 

 \- frontend/src/api/client.ts:3-5

 \- frontend/src/hooks/useStreamChat.ts:23

 

 但生产 nginx 只配置了静态文件路由，没有 /api 反向代理：

 

 \- frontend/Dockerfile:8-10

 

 Compose 里虽然设置了：

 

 \- docker/docker-compose.yml:104-105

 

 VITE_API_BASE_URL=http://localhost:8000/api/v1

 

 但前端代码没有读取这个变量。结果是容器部署后前端请求会打到 nginx 自己的 /api/v1，通常 404。

 

3. 查询接口没有校验 session 和 kb 是否存在/匹配

 

 backend/app/api/endpoints/chat.py:47-53 直接把 kb_id 和 session_id 传给 RAG。

 backend/app/services/rag_service.py:26-30、backend/app/services/rag_service.py:60-63 也没有校验 session 是否属于该知识库。

 

 影响：

 

 \- 可以拿 KB-A 的 session 去问 KB-B。

 \- 聊天历史会被串到错误知识库上下文。

 \- 不存在的 session 可能在写入 message 时变成 500。

 

4. 同名文件删除会误删向量

 

 写入 Milvus 时只保存原始文件名：

 

 \- backend/app/services/ingestion_service.py:33-39

 

 删除时按 document_name == filename 删除：

 

 \- backend/app/db/milvus_client.py:103-108

 

 如果同一个知识库上传两个同名文件，删除其中一个会把两个文件的向量都删掉。应改成按 doc_id 或唯一 stored name 删除。

 

 中优先级

 

5. 文档状态接口没有校验 kb_id

 

 路由是 /knowledge-bases/{kb_id}/documents/{doc_id}/status，但实现里只按 doc_id 查：

 

 \- backend/app/api/endpoints/document.py:99-107

 

 这会导致跨知识库探测文档状态。

 

6. 删除知识库不会删除上传文件

 

 backend/app/services/knowledge_base_service.py:60-65 只删除 SQLite 记录并 drop Milvus collection，没有清理 UPLOAD_DIR 下的原始文件。

 删除知识库后文件仍留在磁盘上。

 

7. 后台文档摄入没有恢复机制

 

 上传后用 fire-and-forget：

 

 \- backend/app/api/endpoints/document.py:70-72

 

 如果进程在摄入中重启，文档可能永久停在 pending / processing。backend/app/main.py:15-30 启动时也没有恢复或标记失败逻辑。

 

8. embedding 加载开启了远程代码执行

 

 backend/app/services/embedding_service.py:20-21：

 

 SentenceTransformer(..., trust_remote_code=True)

 

 而模型名来自环境变量。生产环境这有供应链风险，建议默认关闭，除非明确使用可信模型并 pin 版本。

 

9. 500 错误会把内部异常返回给客户端

 

 backend/app/core/exception_handlers.py:21-31 返回了 str(exc)。

 这可能泄露 SQL 错误、文件路径、第三方 API 信息等。

 

10. Docker 配置里 Milvus standalone 实际没被后端使用

 

 Compose 配了 standalone Milvus：

 

 \- docker/docker-compose.yml:40-58

 \- docker/docker-compose.yml:68-70

 

 但后端代码总是启动 Milvus Lite：

 

 \- backend/app/db/milvus_client.py:16-20

 

 MILVUS_HOST / MILVUS_PORT 没有被使用，所以当前 Docker Milvus 服务基本是摆设。

 

 前端问题

 

11. 切换知识库时会保留旧 session 和 messages

 

 点击知识库只调用：

 

 \- frontend/src/components/KnowledgeBase/KnowledgeBaseList.tsx:58

 

 setActiveKnowledgeBase(kb.id)

 

 但 store 的 setActiveKnowledgeBase 只改 KB ID：

 

 \- frontend/src/store/appStore.ts:46

 

 不会清空 activeSessionId 和 messages。用户切换知识库后可能继续看到/发送上一个知识库的会话。

 

12. 取消流式请求会留下空 assistant 气泡

 

 请求开始时先添加空 assistant：

 

 \- frontend/src/hooks/useStreamChat.ts:17-20

 

 取消时直接 return：

 

 \- frontend/src/hooks/useStreamChat.ts:80-84

 

 然后 isStreaming 变 false，MessageBubble 会渲染一个空回复：

 

 \- frontend/src/components/Chat/MessageBubble.tsx:26-29

 

13. SSE 解析比较脆弱

 

 frontend/src/hooks/useStreamChat.ts:44-77 是逐行看到 data: 就立即 JSON.parse，没有按 SSE 的空行分隔符累计完整 event。当前后端一行 JSON

 通常能跑，但如果以后出现多行 data、最后没有换行、或代理切包，就容易丢 sources / done。

 主要报错包括：

 

 \- frontend/src/api/chat.ts:15 使用 any

 \- frontend/src/components/Chat/ChatPanel.tsx:21 使用 any

 \- frontend/src/components/Chat/ChatPanel.tsx:41 空 catch block

 \- frontend/src/components/Chat/ChatSessionList.tsx:15,26,35 空 catch block

 \- frontend/src/hooks/useStreamChat.ts:80 使用 any
